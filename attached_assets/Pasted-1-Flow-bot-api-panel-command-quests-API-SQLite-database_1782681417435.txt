1. Flow หลักตอนนี้เริ่มมีแล้ว

ตอนนี้ระบบมี:

bot/
api/
panel command
quests API
SQLite database
slash commands

บอท import คำสั่งหลายตัวแล้ว เช่น quest-add, quest-list, quest-done, quest-status, run, stop, panel

และ /panel มีปุ่มควบคุมแล้ว เช่น ดูรายการ, เพิ่ม Quest, Mark Done, สถิติ, Refresh

อันนี้ถือว่าเริ่มเป็น “ระบบ” แล้ว ไม่ใช่โครงเปล่า

2. สิ่งที่ขาดจริง ๆ ตอนนี้
A. ยังไม่มีระบบรันแบบ production ชัดเจน

ตอนนี้มี bot/package.json กับ api/package.json แยกกัน แต่ root repo ยังไม่ได้ทำให้รันง่ายแบบ:

npm run dev:api
npm run dev:bot
npm run start:api
npm run start:bot

ตอน deploy จริง คนจะงงว่า:

ต้องเข้า bot/ หรือ api/ ก่อน?
ต้อง npm install ที่ไหน?
ต้อง start ตัวไหนก่อน?
API URL ต้องใส่อะไร?

ควรเพิ่ม docs หรือ root scripts ให้ชัด

B. Bot กับ API ยังไม่มี health check ฝั่งบอท

API มี /health แล้ว
แต่บอทยังไม่มีคำสั่งเช็กว่า API ต่อได้ไหมจริง ๆ

ควรเพิ่ม:

/api-status

ให้บอทเช็ก:

API online/offline
API latency
database ok ไหม
version อะไร

ไม่งั้นเวลา API ล่ม ผู้ใช้จะเห็นแค่ command error

C. Database ยังเล็กไป

ตอนนี้ database มีแค่ table quests:

id
name
deadline
note
done
done_at
created_at

ถ้าใช้งานจริงควรมีอย่างน้อย:

users
guilds
guild_settings
quests
quest_logs
jobs
reminders
audit_logs

เพราะตอนนี้ quest ทุกอย่างน่าจะอยู่รวมกันหมด ยังไม่แยก server / user / owner ชัด

D. ยังไม่มี multi-guild isolation

ถ้าเอาบอทเข้า 2 server ตอนนี้ระบบควรถามว่า:

Quest นี้เป็นของ server ไหน?
ใครเป็นเจ้าของ?
ใครมีสิทธิ์ลบ?
ใคร mark done ได้?
panel ของ server ไหน?
log channel ของ server ไหน?

ตอนนี้ API quest route ยังเป็น global list เช่น GET /quests ดึงทั้งหมด

ควรเปลี่ยนระบบให้ quest มี:

guild_id
owner_user_id
created_by
assigned_to

ไม่งั้นข้อมูลหลาย server จะปนกัน

E. ยังไม่มี permission system

ตอนนี้ปุ่มใน panel ใครกดก็อาจใช้งานได้ ถ้าไม่มี guard เพิ่ม

ควรมี permission layer:

OWNER_ONLY
ADMIN_ONLY
MANAGER_ROLE
EVERYONE_READ_ONLY

ตัวอย่าง:

เพิ่ม quest = member ใช้ได้
ลบ quest = admin/owner
ตั้งค่า panel/log channel = admin
ดู status = ทุกคน
F. ยังไม่มี guild settings

ควรมีคำสั่ง:

/settings log-channel
/settings panel-channel
/settings manager-role
/settings timezone

และ API endpoint:

GET /guilds/:guildId/settings
PATCH /guilds/:guildId/settings

ตอนนี้ยังไม่เห็นส่วน settings จริงใน API มีแค่ /quests กับ /runner

G. ยังไม่มี Worker / Reminder

ระบบจริงควรมี worker แยก:

เช็ก deadline ทุก X นาที
แจ้งเตือน quest ใกล้หมดเวลา
ส่ง daily summary
ส่ง overdue alert
ส่ง weekly report

ตอนนี้ยังไม่มี worker/ หรือ scheduler ชัดเจน

ถ้าไม่มี worker ระบบจะเป็นแค่ “บันทึกเควส” แต่ยังไม่เป็น “Quest Control System”

H. Job state ยังไม่ควรอยู่ memory อย่างเดียว

ส่วน job/status ถ้าเก็บใน memory พอ API restart ทุกอย่างหาย

ระบบจริงควรเก็บ job ลง database:

job_id
quest_id
guild_id
user_id
status
started_at
updated_at
finished_at
last_error
last_message_id

แล้ว restart แล้วอ่านกลับมาได้

I. API auth ยังเบาไป

ตอนนี้ API เช็ก x-api-secret เฉพาะกรณีมี API_SECRET เท่านั้น ถ้าไม่ได้ตั้ง secret มันจะปล่อยผ่านทั้งหมด

Production ควรเปลี่ยนเป็น:

ถ้า NODE_ENV=production แล้วไม่มี API_SECRET → start ไม่ได้

และควรมี:

rate limit
request size limit
structured error
request id
J. CORS เปิดกว้าง

app.use(cors()) เปิดหมดทุก origin

ตอน dev ไม่เป็นไร แต่ production ควรล็อก origin เช่น dashboard domain เท่านั้น

K. Bot commands บางตัวอาจยังต้องเช็ก await

ใน panel ใหม่ใช้ await getStats(), await getAllQuests(), await addQuest() แล้ว ดีขึ้นมาก

แต่คำสั่งแยกอย่าง quest-add, quest-list, quest-done, quest-remove, quest-status จากรอบก่อนยังควรเช็กว่าปรับ await ครบหรือยัง เพราะถ้าไม่ครบจะพังเวลาเรียก API ที่ return Promise

L. ยังไม่มี CI สำหรับ bot/api

CI ตอนนี้ยังเน้น frontend เดิม เช่น install root, i18n check, test, build frontend

ควรเพิ่ม job ใหม่:

bot-check
api-check

เช่น:

cd bot
npm ci
node --check src/index.js
node --check src/register-commands.js
cd api
npm ci
node --check src/index.js
M. ยังไม่มี deployment guide

ต้องมีเอกสารบอกชัด:

API deploy ที่ไหน
Bot deploy ที่ไหน
ต้องตั้ง ENV อะไร
ต้อง register commands ตอนไหน
start command คืออะไร
database เก็บถาวรไหม

ตอนนี้ README ยังเป็นของ Tauri app เดิม ไม่ใช่ bot/api system ใหม่

3. ENV ที่ควรจัดใหม่
bot env ควรมี
DISCORD_BOT_TOKEN=
DISCORD_CLIENT_ID=
DISCORD_GUILD_ID=
OWNER_ID=
API_URL=
API_SECRET=
TIMEZONE=Asia/Bangkok

ตอนนี้ bot/.env.example มีของพวกนี้แล้ว ถือว่าดี

api env ควรมีเพิ่ม

ตอนนี้มี:

PORT=3000
DATABASE_PATH=./data/quests.db
API_SECRET=changeme
DISCORD_BOT_TOKEN=

ควรเพิ่ม:

NODE_ENV=production
PUBLIC_BASE_URL=
CORS_ORIGIN=
LOG_LEVEL=info
TIMEZONE=Asia/Bangkok

5. Checklist ที่ควรทำต่อ

เรียงตามความสำคัญจริง:

1. ตรวจ commands แยกทั้งหมดให้ await API ครบ
2. เพิ่ม /api-status
3. เพิ่ม permission guard
4. เพิ่ม guild_settings table
5. เพิ่ม guild_id / owner_user_id ใน quests
6. เพิ่ม quest_logs table
7. เพิ่ม reminder/worker
8. เพิ่ม CI bot/api
9. เพิ่ม deployment docs
10. เขียน README ใหม่แยก Desktop App vs Bot/API
6. สถานะสุดท้าย
พร้อมเป็น prototype: ใช่
พร้อมให้เพื่อนช่วยเขียนต่อ: ใช่
พร้อม deploy จริง: ยัง
พร้อมเป็นระบบหลาย server: ยัง
พร้อมเป็น dashboard/API ระยะยาว: ยังขาด schema + worker + docs
